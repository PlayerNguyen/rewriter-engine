# Plan: Orval REST Client Generation Pipeline

## Purpose

Generate a fully type-safe REST client from the server's OpenAPI spec, packaged as `@rewriter/rest-client` for consumption by any workspace package (dashboard, future `table-ui`, etc.). Removes the need for manual fetch calls, type stubs, or URL string duplication across consumers.

## Investigation Summary

### Current State

- `apps/server` uses `hono-openapi` with `openAPISpecs` middleware — spec generated **per-request** at `/api/v1/doc`
- Routes use `describeRoute` + `hono-openapi/zod` resolver for schema documentation
- Scalar UI at `/api/v1/scalar` for human-readable docs
- No persisted spec file → no existing client generation

### Orval Capabilities (from docs)

| Feature | Detail |
|---|---|
| Input | Static JSON/YAML file OR live URL |
| HTTP clients | `fetch`, `axios`, or custom mutator |
| Output modes | `single`, `split`, `tags-split` (one file per OpenAPI tag) |
| Data fetching | React Query, Vue Query, SWR, Solid Query, Angular Query hooks |
| Validation | Zod / Effect schema generation alongside types |
| Mocking | MSW handler generation for testing |
| Hooks | `after` transforms for post-processing generated files |
| Framework | Hono server-side client generation |

### hono-openapi Persisting (from docs)

`generateSpecs(app, options)` returns the OpenAPI spec object — can be written to JSON or YAML. This is the bridge between the live Hono app and Orval's static-file input.

```typescript
import fs from 'node:fs';
import { generateSpecs } from 'hono-openapi';

const specs = await generateSpecs(mainApp, { /* documentation metadata */ });
fs.writeFileSync('./openapi.json', JSON.stringify(specs, null, 2));
```

## Scope

| In scope | Out of scope |
|---|---|
| `apps/server/scripts/generate-openapi.ts` — persist spec to file | Migrating existing endpoints to `describeRoute` (already done) |
| `apps/server/openapi.json` — committed artifact | Adding new API endpoints |
| `packages/rest-client/` — package with Orval config, mutator, barrel | Implementing React Query hooks (future follow-up) |
| Custom fetch mutator with auth, error normalization, env-based baseURL | Implementing MSW mocks (future follow-up) |
| Root-level `generate:client` script | CI integration (future follow-up) |
| `tags-split` output mode for tree-shakeable imports | Zod schema generation (future follow-up) |

## Architecture

```
apps/server/
├── scripts/
│   └── generate-openapi.ts             ← generateSpecs(app) → write openapi.json
├── openapi.json                        ← COMMITTED (generated artifact, version-controlled)
└── src/
    └── app.ts                          ← unchanged (spec generation is external, not in-app)

packages/rest-client/
├── package.json                        ← name: @rewriter/rest-client
├── tsconfig.json
├── orval.config.ts                     ← reads ../../apps/server/openapi.json
├── src/
│   ├── index.ts                        ← barrel: re-exports generated api + mutator
│   ├── mutator/
│   │   └── fetch-instance.ts           ← custom fetch with env baseURL, auth, error handling
│   └── generated/                      ← gitignored, fully generated by Orval
│       ├── api/                        ← tags-split → /health, /Table, etc.
│       └── model/                      ← shared type schemas (DTOs, responses)
```

### Data Flow

```
apps/server/                          packages/rest-client/
─────────────                         ──────────────────────
1. Dev changes routes
   (describeRoute, schemas)
          │
2. bun run generate:spec ──────────→  openapi.json
                                          │
3. bun run generate:client ──────────→  orval reads openapi.json
                                          │
4.                                       src/generated/api/**  ← fully typed fetch functions
                                         src/generated/model/** ← DTO types
                                          │
5. Consumer (dashboard, table-ui, etc.)
     import { getHealth } from '@rewriter/rest-client/api/health'
     const result = await getHealth()
```

## Key Design Decisions

| Decision | Rationale |
|---|---|
| `fetch` over `axios` | Zero extra deps; works in browser, Bun, and Node; interceptable via custom mutator; matches the monorepo's minimal-dependency philosophy |
| `tags-split` mode | Consumers import only what they need (`@rewriter/rest-client/api/health`); tree-shakeable; no monolithic barrel |
| Custom mutator | Single source of truth for baseURL, auth headers, error normalization; shared by every generated API function |
| Spec committed to repo | CI and devs always have the source of truth; no need to run the server to generate the client; diffable in PRs |
| `openapi.json` in `apps/server/` | Belongs with the source of truth (the server); co-located with the code that defines the API |
| Orval config in `packages/rest-client/` | Client config belongs with the output package; other consumers could have their own Orval configs targeting different specs |
| `API_URL` env var for baseURL | Works in both browser (`VITE_API_URL`) and server contexts; no hardcoded URLs |

## File Plan

### New: `apps/server/scripts/generate-openapi.ts`

```typescript
import fs from 'node:fs';
import path from 'node:path';
import { generateSpecs } from 'hono-openapi';
import { createApp } from '../src/app';

async function main() {
  const app = createApp();

  const specs = await generateSpecs(app, {
    documentation: {
      info: {
        title: 'Rewriter API',
        version: '0.0.1',
        description: 'API for the content rewriter pipeline',
      },
      servers: [{ url: 'http://localhost:3001', description: 'Local Dev' }],
    },
  });

  const outputPath = path.resolve(import.meta.dirname, '../openapi.json');
  fs.writeFileSync(outputPath, JSON.stringify(specs, null, 2));
  console.log(`OpenAPI spec written to ${outputPath}`);
  console.log(`  Paths: ${Object.keys(specs.paths ?? {}).length}`);
}

main();
```

### New: `packages/rest-client/src/mutator/fetch-instance.ts`

```typescript
export type RequestParams = {
  url: string;
  method: 'GET' | 'POST' | 'PUT' | 'DELETE' | 'PATCH';
  headers?: Record<string, string>;
  body?: unknown;
};

export const customFetchInstance = async <T>(params: RequestParams): Promise<T> => {
  const baseURL = process.env.API_URL ?? 'http://localhost:3001';

  const headers: Record<string, string> = {
    'Content-Type': 'application/json',
    ...params.headers,
  };

  // Inject auth token if available (browser-side)
  if (typeof window !== 'undefined') {
    const token = localStorage.getItem('token');
    if (token) {
      headers['Authorization'] = `Bearer ${token}`;
    }
  }

  const res = await fetch(`${baseURL}${params.url}`, {
    method: params.method,
    headers,
    body: params.body ? JSON.stringify(params.body) : undefined,
  });

  if (!res.ok) {
    const error = await res.json().catch(() => ({}));
    throw { status: res.status, ...error };
  }

  return res.json();
};
```

## Scalability

| Dimension | Mechanism |
|---|---|
| **New APIs from the same server** | Already covered — `tags-split` splits every documented route into its own module under `src/generated/api/` |
| **New apps exposing OpenAPI** | Each app adds its own `scripts/generate-openapi.ts` → `openapi.json`. Add a new named section to `orval.config.ts` (e.g., `explorer: { input: '../../apps/explorer/openapi.json', output: ... }`) |
| **Multiple consumer packages** | `tags-split` + `exports` map ensures each consumer imports only the tags it needs. Example: dashboard imports `/health` + `/Table`; future mobile app imports only what it calls |
| **Different HTTP flavors per consumer** | Orval supports multiple output configs from the same spec. Add a `rewriter-react-query` section with `client: 'react-query'` → outputs to `src/generated/react/` — opt-in per consumer via import path |
| **Large API surface** | `tags-split` avoids monolithic files; each tag is its own module; no consumer pays for code it doesn't import |

## Extensibility

| Capability | Implementation |
|---|---|
| **Auth** | Mutator reads token from `localStorage` (browser) or `Authorization` header (server). Swap logic in single file. |
| **Error normalization** | Mutator catches non-OK responses, parses JSON, throws structured `{ status, ...body }`. Consumers get consistent error shapes. |
| **Request transforms** | Mutator can camelCase/decapitalize request bodies before sending, or snake_case responses after receiving. |
| **Response transforms** | Orval `override.response.transformer` or mutator post-processing. |
| **Zod validation** | Add `zod: true` to output config → Zod schemas generated alongside types in `src/generated/model/`. Consumers can validate at runtime. |
| **MSW mocks** | Add `mock: true` → Orval generates MSW handlers for testing. |
| **Base URL from env** | `API_URL` env var at runtime; works with Vite (`VITE_API_URL`), Bun, Docker, CI. |
| **Pre-commit hook** | Add `bun run generate:client` to pre-commit → CI fails if generated client is out of date with committed spec. |
| **Orval Docker** | For CI environments without Node 22.18+: use `ghcr.io/orval-labs/orval` Docker image. |
| **Hooks** | `after: { all: [addUseClientDirective] }` for React Server Components compatibility; custom formatters. |

## Execution Order

1. Create `apps/server/scripts/generate-openapi.ts` using `generateSpecs(app, ...)`
2. Add `"generate:spec"` to `apps/server/package.json`
3. Run `bun run --filter @rewriter/server generate:spec` → commit `openapi.json`
4. Create `packages/rest-client/package.json` (match `@rewriter/table-core` pattern)
5. Create `packages/rest-client/tsconfig.json`
6. Create `packages/rest-client/src/mutator/fetch-instance.ts`
7. Create `packages/rest-client/orval.config.ts`
8. Add `@rewriter/rest-client` path alias to root `tsconfig.json`
9. Add `packages/rest-client/src/generated/` to `.gitignore`
10. Run `bun install` to link workspace
11. Run `bun run --filter @rewriter/rest-client generate` to generate client
12. Create `packages/rest-client/src/index.ts` barrel
13. Run `bun run --filter @rewriter/rest-client typecheck`
14. Add root scripts: `generate:spec`, `generate:client`
